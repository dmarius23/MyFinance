1. Executive Summary

MyFinance is a multi-tenant SaaS portal connecting accounting firms with the client companies (legal entities) whose accounting they manage. It centralizes document exchange, financial reporting, fiscal-declaration handling, payroll distribution, notifications, and internal task tracking in one secure system.

Its two differentiators versus the firm's current tooling are client-driven document upload with automatic detection of missing documents (reconciled against the bank statement), and automatic extraction of the amounts owed to the state from fiscal declaration PDFs — replacing a manual data-entry form.

The product is designed to be sold to multiple accounting firms, so multi-tenancy, per-tenant isolation, audit, and GDPR-compliant data handling are first-class concerns.

2. Goals & Objectives

  • Let client representatives upload source documents (bank statements, invoices, receipts) and see, in real time, what is still missing for the period.
  • Automatically extract amounts owed to the state from fiscal declarations and send templated payment emails with deadlines and the correct treasury accounts.
  • Give accounting-firm employees a single dashboard showing every client company's monthly status (documents, reports, taxes, payroll).
  • Centralize financial-statement and payroll distribution with per-month sent-status tracking.
  • Provide configurable notifications (email + in-app) for missing documents and period deadlines.
  • Operate as a secure, auditable, GDPR-compliant multi-tenant SaaS sellable to many accounting firms.

5. Phasing Strategy

Requirements are tagged by phase to indicate delivery priority.

Proof of Concept (PoC)

  • Extract amounts per tax type from ANAF fiscal-declaration PDFs (structured parse with OCR/manual fallback).
  • Parse a text-based bank-statement PDF into transactions.
  • OCR a receipt photo and extract amount + counterparty.
  • Reconcile uploaded invoices/receipts against the bank statement and flag missing documents.
  • Auto-fill a state-payment email (amounts + deadline + treasury account).
  • Minimal authentication and company setup to support the above.

Minimum Viable Product (MVP)

  • MOD-01 to MOD-12 in full (multi-tenant).
  • Bounded representative-facing chatbot + knowledge base (MOD-13).
  • Responsive web and installable PWA with camera capture and web push (MOD-14).
  • Authentication: email+password, Google OAuth, TOTP MFA for staff.
  • Notifications via email + in-app.

Future Releases

  • WhatsApp Business API notification channel.
  • e-Factura / ANAF SPV integration as a primary invoice source.
  • In-app two-way messaging between firm and representatives.
  • Native mobile application.
  • Per-employee payslip distribution and automated payslip/tax-document distribution.
  • Payroll sync agent or direct payroll-system integration.
  • Per-tenant template customization.
  • Per-employee client-company access restriction.
  • Database-per-tenant isolation as a premium tier.

6. Functional Requirements

6.1 Tenant Administration

IDRequirementPhase
FR-001The system shall allow a platform super-admin to create, view, edit, suspend, reactivate, and archive accounting-firm tenants.MVP
FR-002The system shall record per tenant: firm name, fiscal code (CUI), primary contact, status, plan/tier, and creation date.MVP
FR-003On tenant creation, the system shall provision an initial tenant-admin account and send an invitation email to set a password.MVP
FR-004The system shall let the super-admin set per-tenant limits and feature flags (max users, max client companies, and toggles for chatbot, WhatsApp, and email-agent).MVP
FR-005The system shall enforce tenant data isolation so no user can access another tenant's data.MVP
FR-006The system shall allow optional per-tenant branding (firm name and logo) for a white-label appearance.MVP
FR-007The system shall record all super-admin actions in the audit log.MVP

Business Rules

  • Suspending a tenant disables all its users' access and pauses its automated jobs (scheduled notifications, email-ingestion agent) but retains data.
  • The tenant administration module is visible only to platform super-admins.
  • Plan limits are hard-enforced: actions that would exceed a limit are blocked.
  • Archiving or deleting a tenant triggers a GDPR purge that requires explicit confirmation and respects the legal retention window.
  • Super-admins cannot impersonate or enter tenant data in MVP.

Workflows

Onboard a firm: super-admin creates tenant -> sets plan, limits, feature flags -> invites tenant admin -> tenant admin completes firm setup.

Offboard a firm: suspend -> grace period (data retained) -> optional archive/purge.

Edge Cases & Error Handling

  • Expired or re-sent tenant-admin invitation.
  • A suspended tenant must not fire scheduled emails or run its email-ingestion agent.
  • Tenant identifier/subdomain uniqueness conflict.
  • An action attempts to exceed a hard plan limit (blocked).

State Transitions

  • Provisioned -> Active: tenant created and admin onboarded.
  • Active -> Suspended: super-admin suspends (access disabled, data retained).
  • Suspended -> Active: super-admin reactivates.
  • Active/Suspended -> Archived/Purged: offboarding after retention rules.

6.4 Document Intake & Reconciliation

IDRequirementPhase
FR-021The system shall accept document uploads (PDF or photo) via three channels: representative portal upload, employee upload on behalf of a company, and an email-ingestion agent that imports attachments from the firm's mailbox (Gmail/M365 via OAuth).MVP
FR-022The system shall classify each document (bank statement / invoice / receipt) and associate it with a company and a period (month or quarter per the company's VAT cadence).MVP
FR-023The system shall parse the bank statement into transactions and reconcile them against uploaded invoices/receipts, identifying transactions with no supporting document.MVP
FR-024The system shall show the uploader a real-time completeness view of what is still missing for the period.MVP
FR-025The system shall maintain a per-company, per-period status (Not started / Partial / Complete) used by dashboards and reminders.MVP
FR-026The email-ingestion agent shall route documents it cannot confidently assign to a company into a manual triage queue.MVP
FR-026bThe system shall auto-match high-confidence bank-to-document pairs and queue low-confidence or ambiguous matches for human confirmation.MVP
FR-026cFor each bank transaction the system shall decide whether a supporting invoice/receipt is required and record the decision source as SYSTEM_RULE (deterministic base rule), LEARNED_RULE (adaptive rule learned from accountant decisions), or ACCOUNTANT_SET (manual). No LLM is used; classification is fully deterministic and explainable.MVP
FR-026dThe system shall apply deterministic base rules using structured signals: incoming/credit amounts are income (no document); payments to Treasury IBANs (CAM, contributii, impozit, TVA) are covered by declarations; transfers between the company's own accounts need no document; salary descriptions are covered by payroll; leasing payments require the leasing invoice; other supplier debits require an invoice/receipt.MVP
FR-026eThe system shall let an accountant override any transaction's document-requirement flag (e.g. mark 'no document needed' with an optional reason); the manual override always takes precedence and is recorded in the audit log.MVP
FR-026fWhen an accountant sets a document-requirement decision, the system shall learn an adaptive rule for that company keyed on the counterparty account (IBAN) and normalized description, and shall automatically apply that decision to matching transactions in later periods so the accountant is not asked again.MVP
FR-026gThe system shall drive the missing-documents list and reminder emails only from transactions that require a document and remain unmatched.MVP

Business Rules

  • A period is Complete when the bank statement is uploaded and every transaction requiring a supporting document has a matched one.
  • Cadence (monthly/quarterly) follows the company's VAT period.
  • Duplicate uploads (same invoice twice) are detected.
  • Cash receipts that legitimately never appear on the bank statement are supported.
  • Imports the agent cannot map go to triage, not silently dropped.
  • Document-requirement decisions follow precedence: a manual accountant decision wins over a learned rule, which wins over a base rule. The system adapts to each firm's corrections over time (no LLM).
  • Only transactions that require a document and remain unmatched contribute to the missing-documents list and trigger client reminders.

Workflows

Representative flow: upload bank statement + receipts/invoices -> extraction -> reconciliation -> missing items shown live -> representative adds missing -> status becomes Complete.

Email-agent flow: email arrives -> attachments pulled -> classified -> matched to company (or triage) -> same reconciliation pipeline.

Edge Cases & Error Handling

  • Blurry photo or low-confidence extraction routed to manual review.
  • Multiple bank accounts or multi-page statements per company per month.
  • Amount matches but date differs -> suggested match needing confirmation.
  • Foreign-currency transactions.
  • Employee manually matching or marking a transaction as 'no document needed'.

State Transitions

  • CompanyPeriod: Empty -> Partial -> Complete -> (Reopened).
  • Document: Uploaded -> Extracted -> Matched | Unmatched -> (Confirmed).

6.5 Document Extraction Engine

IDRequirementPhase
FR-027The system shall extract the amounts to pay per tax type, plus period and deadline, from ANAF smart-declaration PDFs by reading the embedded declaration XML (e.g. the d212.xml inside a D212), parsing the named fields directly (e.g. impozit_venit, CASS, dif_de_plata).MVP
FR-027aThe system shall support multiple ANAF declaration types (e.g. D212, D300, D301, D112, D100) via a per-declaration-type XML mapper behind a single extractor interface, so new declaration schemas can be added without changing the rest of the system.MVP
FR-027bThe system shall validate extracted declaration amounts against the declaration's own total field (e.g. component taxes must sum to dif_de_plata) and flag any mismatch for review before the figures can drive a payment email.MVP
FR-028The system shall extract from invoices (text PDF or XML) the counterparty (furnizor/debitor), counterparty IBAN, date, due date, total, VAT, and document number; for photographed receipts it shall use OCR and return a per-field confidence score.MVP
FR-029The system shall parse readable bank-statement PDFs into normalized transactions (date, amount, direction, counterparty, counterparty IBAN, reference, running balance) using a per-bank parser behind a single BankStatementParser interface, so additional banks (BRD, BT, BCR, ING, etc.) can be added without changing reconciliation.MVP
FR-030The system shall extract trial-balance / profit-situation data (account code, account name, amounts) from readable PDFs and cross-check the parsed lines against the document's stated totals before generating a report.MVP
FR-031The system shall route any field below a confidence threshold (OCR) or any failed cross-check (structured documents) to human review, and feed corrections back.MVP
FR-032The system shall support Romanian (diacritics) and RON plus foreign currencies.MVP

Business Rules

  • Structured/electronic sources are always preferred over OCR: ANAF declarations are read from embedded XML; invoices from XML or text PDF; bank statements and balances from readable-PDF text. OCR is used ONLY for photographed receipts.
  • All declarations and bank statements provided to the system are readable (machine-text) PDFs, not scanned images; declaration types and bank layouts vary, handled by pluggable per-type/per-bank parsers.
  • Structured extractions are validated by an internal cross-check (declaration component taxes sum to the stated total; balance lines sum to the stated totals) before the values may post or drive an email.
  • Nothing below the OCR confidence threshold, and nothing that fails a cross-check, is auto-committed.
  • Each extraction is audited: source document -> extracted values -> reviewer.

Workflows

Detect type and format -> pick extractor (structured parse vs OCR/AI) -> extract -> confidence check -> auto-accept or queue review -> persist normalized data for the consuming module.

Edge Cases & Error Handling

  • Low-resolution or rotated photos.
  • Multi-page or handwritten receipts.
  • A declaration that is a flat scan with no embedded data (OCR fallback).
  • An unsupported bank layout (manual-entry fallback).
  • Mixed-language documents.

State Transitions

  • ExtractionJob: Pending -> Extracted -> (Needs Review -> Reviewed) -> Committed.

6.7 Fiscal Declarations & State Payments

IDRequirementPhase
FR-038The system shall let an employee upload the monthly fiscal declaration PDF(s) per company, as separate files per tax.MVP
FR-039The system shall extract the amount owed per tax type plus period and deadline, populating a monthly tax summary (impozit micro/profit/salarii, TVA, contributii sociale, CAM, chirie, dividende).MVP

State Transitions

  • MonthlyTaxObligation: NoDeclaration -> Extracted -> Verified -> EmailSent -> (Resent).

6.12 Audit, Security & GDPR

IDRequirementPhase
FR-062The system shall maintain an immutable audit log of all actions by representatives, employees, super-admins, and automated processes, queryable by an admin.MVP
FR-063The system shall let an employee delete a client company's data on the representative's request, following the retention-aware process and recorded in the audit log.MVP
FR-064The system shall encrypt sensitive identifying data at rest and enforce TLS/HTTPS in transit.MVP
FR-065The system shall enforce tenant isolation and role scoping and record access-control changes.MVP

Business Rules

  • Retention-aware deletion: revoke access and anonymize non-required personal data immediately; retain legally-required accounting records for the retention window; full purge after expiry.
  • Audit entries are immutable and retained per policy.
  • Accounting-records retention obligations take precedence over immediate erasure.

Workflows

Every action writes an audit entry.

Deletion request -> employee initiates -> confirmation -> anonymize/purge per retention rules -> audit recorded.

Edge Cases & Error Handling

  • Erasure-vs-retention conflict.
  • Audit-log volume and retention.
  • Exporting audit for an authority.

State Transitions

  • DeletionRequest: Requested -> Anonymized -> (Retention window) -> Purged.

10. Assumptions & Constraints

The following are assumed to be true:

  • Bank statements are provided as readable (machine-text) PDFs, not scanned images. Layouts vary by bank (BRD, BT, BCR, ING, etc.) and are handled by pluggable per-bank parsers. (Validated on a real BRD statement.)
  • Invoices are available as structured PDF or XML; only photographed receipts require OCR. (Validated on a real BT Leasing invoice — supplier, IBAN, total, due date.)
  • Payroll is computed in an external system and uploaded manually.
  • Fiscal declarations are ANAF smart PDFs carrying embedded declaration XML; amounts are read from named XML fields and validated against the declaration's own total. Multiple declaration types (D212, D300, D301, D112, ...) are supported via per-type XML mappers. (Validated on a real D212 — total to pay 10,520 RON = 800 income tax + 9,720 CASS.)
  • The trial-balance / profit-situation is a readable PDF whose parsed lines cross-check to stated totals. (Validated on a real Situație profit — class 6/7 sums matched the document to the cent.)
  • Standard Romanian payment deadlines apply unless overridden per company/month.
  • The product is operated as multi-tenant SaaS hosted in an EU region.
