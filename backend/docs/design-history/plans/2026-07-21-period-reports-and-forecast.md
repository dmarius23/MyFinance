# Period Reports + Forecast (MOD-06, rep PWA) Implementation Plan

> **For agentic workers:** subagent-driven-development. Steps use `- [ ]` checkboxes.

**Goal:** Let a company representative (and, later, firm staff) view and download the financial
report + charts for a **month, calendar quarter, calendar half-year, or calendar year**, plus a
**simple trend projection** of the coming months. Monthly reports already exist (generated from the
accountant's monthly trial-balance / *balanță* upload); this adds server-side **period aggregation**
(quarter/half/year), a **rep-facing read path**, and an **ephemeral forecast**.

**Architecture:** Extend `ro.myfinance.reports`. Aggregation is **server-side and shared** by both
audiences — the PWA and the firm hub are thin clients over the same endpoints. Period reports are
**persisted and cached** (new `period_report` table) keyed by a **source fingerprint** so a re-uploaded
month auto-invalidates the enclosing quarter/half/year. The forecast is **computed per request and
never stored** (it is volatile and non-authoritative). A worker job pre-warms closed calendar periods
(optimization only; correctness comes from the fingerprint check).

**Tech Stack:** Java 21 / Spring Boot 3 / JPA / Flyway / PDFBox / Testcontainers; React 18 / TS / Vite
/ TanStack Query / react-i18next.

## Confirmed decisions

- **API shape:** explicit `granularity` + `period` pair. `period` stays the existing ISO-date anchor
  (any day in the period; callers send the first day); `granularity ∈ {MONTH, QUARTER, HALF, YEAR}`,
  default `MONTH` (fully backward-compatible).
- **Period data:** aggregate from the existing **monthly** snapshots — no extra uploads.
- **Storage:** *mixture* — closed calendar periods pre-generated by a scheduler; the in-progress /
  not-yet-viewed periods generated on demand and then persisted. One aggregation path; the scheduler
  only warms the cache.
- **Forecast:** simple trend projection (OLS linear fit), clearly labeled non-authoritative, **not stored**.
- **HALF = calendar** H1 (Jan–Jun) / H2 (Jul–Dec). **YEAR = calendar** year (Jan–Dec). No off-calendar
  fiscal year.
- **Balance sheet for a period = the latest present month's closing position** (period-end), not summed
  and not averaged. P&L flows are summed; KPIs recomputed.
- **First milestone audience: the company representative in the PWA.** The firm hub reuses the same
  building blocks afterwards.

## Context / invariants

- **Multi-tenant isolation is invariant #1.** New table gets RLS `ENABLE` + `FORCE`, fail-closed policy.
  New rep read path ⇒ **mandatory cross-tenant AND cross-company isolation tests** (rep A must not read
  company B).
- **Reps are read/download only here** — no upload, no email send. Monthly trial balances still come
  from the accountant; that is why period reports carry an **incomplete ("N of M months")** signal.
- **Extracted/derived amounts are non-authoritative** (golden rule #3). The forecast is a view-layer
  projection: flagged `projected: true`, drawn dashed, never persisted, never fed into a money figure
  or email.
- Existing today (do not regress):
  - `ReportController` — `@PreAuthorize("hasAnyRole('TENANT_ADMIN','EMPLOYEE')")`, firm-only. Endpoints:
    `/api/v1/reports`, `/companies/{id}/report`, `/report/trend`, `/report/pdf`, `/report-emails`,
    `/report/email-body`. `period` is an ISO date.
  - `ReportService.report/summary/trend/ingest`; `ReportCalculator.compute(TrialBalanceData)` (pure);
    `ReportData` (record: `profitLoss` flows, `balanceSheet` closing balances, `kpis`);
    `ReportSnapshot` (table `trial_balance`, one row per company/month, `content_period` guard).
  - `ReportPdfGenerator.generate(ReportData)` and `ReportChartImage` already take a `ReportData` — they
    work unchanged on an aggregated one.
  - Frontend `reportsApi` (`frontend/src/api/reports.ts`), chart components
    (`frontend/src/components/reportCharts.tsx`), `ReportChartsModal`, firm `Reports.tsx` page.
- Built on branch `feat/period-reports-forecast` (worktree off commit `ebe1069`, where the latest committed
  migration is `V37`). Mine is numbered `V43` to stay clear of a parallel branch's uncommitted `V38` — the
  number may need reconciling at merge time. Docker not guaranteed locally → `*IT` may skip; verify
  `mvn -B -DskipTests test-compile` + `mvn -B test` (unit) and FE `npm run lint && npm run build`.

## Task 1: `PeriodAggregator` (pure logic) + unit tests

**Files:** `backend/.../reports/application/PeriodAggregator.java`, `Granularity.java` (enum), tests.

- [ ] **Step 1:** `enum Granularity { MONTH, QUARTER, HALF, YEAR }` with `bounds(LocalDate anchor)`
      returning the enclosing calendar `[start, endExclusive)` and the list of first-of-month dates it
      spans (`monthsExpected`).
- [ ] **Step 2:** `PeriodAggregator.aggregate(List<MonthlyReport>)` where each input carries
      `periodMonth`, `balanced`, and `ReportData`. Rules:
  - P&L (`revenue`, `operatingExpenses`, `grossProfit`, `incomeTax`, `netProfit`) — **sum**;
    `revenueItems`/`expenseItems` — sum by `code` (preserve label; stable order).
  - Balance sheet — take the **latest present month's** `balanceSheet` verbatim (period-end position).
  - KPIs — **recompute** from summed P&L + period-end balance sheet (reuse the `ReportCalculator` ratio
    helpers; extract them if needed). Never average.
  - `balanced` — true only if **all** present months are balanced.
  - `periodStart/periodEnd` — the period bounds; `companyName/cui` from the latest present month.
  - Return the aggregated `ReportData` plus `complete`, `monthsPresent`, `monthsExpected`.
- [ ] **Step 3:** `MONTH` short-circuits to the single input unchanged.
- [ ] **Step 4:** Unit tests: sum-of-flows, period-end balance sheet, recomputed KPIs, all-balanced
      rule, incomplete period (missing middle month), single-month passthrough. No DB.

## Task 2: `V46__period_report.sql` migration (+ RLS + isolation test)

**File:** `backend/src/main/resources/db/migration/V46__period_report.sql`

- [ ] **Step 1:** Table:
```sql
CREATE TABLE period_report (
    id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id        uuid NOT NULL REFERENCES tenant(id),
    company_id       uuid NOT NULL REFERENCES company(id),
    granularity      text NOT NULL,           -- QUARTER | HALF | YEAR (MONTH not stored)
    period_start     date NOT NULL,           -- first day of the calendar period
    report_json      text NOT NULL,
    complete         boolean NOT NULL,
    months_present   int NOT NULL,
    months_expected  int NOT NULL,
    source_fingerprint text NOT NULL,         -- hash of [(period_month, version, content_period)...]
    generated_at     timestamptz NOT NULL DEFAULT now(),
    updated_at       timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT uq_period_report UNIQUE (tenant_id, company_id, granularity, period_start)
);
CREATE INDEX idx_period_report_company ON period_report(tenant_id, company_id, granularity, period_start);
ALTER TABLE period_report ENABLE ROW LEVEL SECURITY;
ALTER TABLE period_report FORCE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON period_report
    USING  (tenant_id = nullif(current_setting('app.tenant_id', true), '')::uuid)
    WITH CHECK (tenant_id = nullif(current_setting('app.tenant_id', true), '')::uuid);
```

      (`findByCompanyIdAndGranularityAndPeriodStart`).
- [ ] **Step 3:** Cross-tenant isolation IT (tenant B cannot see tenant A's `period_report`).

## Task 3: `PeriodReportService` (fingerprint cache) + generalize endpoints

**Files:** `backend/.../reports/application/PeriodReportService.java`, `ReportController`, `ReportService`.

- [ ] **Step 1:** `fingerprint(List<ReportSnapshot>)` = stable hash of sorted
      `(period_month, version, content_period)`.
- [ ] **Step 2:** `PeriodReportService.report(companyId, granularity, anchor)`:
  1. `MONTH` → delegate to existing `ReportService.report`.
  2. Load constituent monthly snapshots for the period; compute current fingerprint.
  3. If stored row exists and fingerprint matches → deserialize + return.
  4. Else → `PeriodAggregator.aggregate(...)`, upsert (json + fingerprint + completeness), return.
  - Carries `complete`/`monthsPresent`/`monthsExpected` out to the web layer.
- [ ] **Step 3:** Add `granularity` (default `MONTH`) to `GET /companies/{id}/report` and
      `/report/pdf`; route through `PeriodReportService`. PDF path feeds the aggregated `ReportData`
      into the unchanged `ReportPdfGenerator`; period-aware filename (e.g. `raport-2026-Q2.pdf`).
- [ ] **Step 4:** Web response exposes `complete`/`monthsPresent`/`monthsExpected` (new response
      wrapper, or headers) so the UI can render the incomplete badge.
- [ ] **Step 5:** Tests: quarter served from cache on 2nd call; re-uploaded month flips fingerprint →
      regenerated; incomplete period reported honestly.
